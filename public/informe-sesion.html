Informe de Sesión de Desarrollo

Resumen de los cambios implementados en la sesión: filtro de período en el embudo de conversión, auditoría completa del flujo de registro y corrección de fallos silenciosos, eventos de onboarding y consolidación de analytics.

1
Selector de período en el embudo de conversión

El embudo de conversión del panel de administración mostraba siempre datos históricos totales, sin posibilidad de acotar el análisis a un intervalo de tiempo concreto. Se ha implementado un selector con cuatro modos de filtrado.

Opciones disponibles

Hoy
00:00 → ahora
7 días
Default · últimos 7d
30 días
Último mes
Personalizado
Desde / hasta (datepickers)

Arquitectura de la solución

Base de datos
Migración SQL — get_funnel_stats(since, until)
La función PostgreSQL acepta ahora dos parámetros opcionales timestamptz. Todos los contadores del embudo, la agregación by_tool y la serie temporal daily filtran por el rango si se proporcionan.
Edge Function — admin-stats
Soporte de since, until y funnelOnly
Acepta POST body con los tres parámetros. El modo funnelOnly: true omite las queries de usuarios y KPIs, devolviendo solo el embudo. Esto hace que el cambio de período sea instantáneo sin recargar toda la tabla de usuarios.
Frontend — AdminDashboard
Componente FunnelDateBar + fetchFunnel()
Al cambiar de período, fetchFunnel() llama al edge function con funnelOnly: true. El panel se atenúa con opacidad durante la carga. fetchData() (recarga completa vía botón Refresh) también usa el período activo.
2
Auditoría del flujo de conversión

Se realizó un análisis exhaustivo del flujo completo desde que el usuario pulsa "Analizar Gratis" hasta completar el registro, revisando todos los estados, llamadas a la API, eventos de analytics y posibles puntos de bloqueo.

Fallos silenciosos críticos encontrados y corregidos

Crítico · Corregido
Edge function devuelve error → usuario bloqueado sin evento
Si signup-instant devolvía un error, el modal quedaba bloqueado mostrando el mensaje de error, pero no se registraba ningún evento de analytics. Era imposible saber cuántos intentos de registro fallaban ni por qué.

Fix: Añadido register_failed { method, error } inmediatamente antes del return de error.
Crítico · Corregido
Login post-signup falla → usuario creado en DB pero sin sesión
El edge function creaba el usuario correctamente ({ created: true }), pero el signInWithPassword posterior podía fallar por latencia o condición de carrera. El usuario quedaba registrado en la base de datos pero incapaz de iniciar sesión, sin ningún evento que reflejase el estado parcial.

Fix: Añadido register_partial_failure { step: 'auto_login' } en el catch de ese login.
Crítico · Corregido
signup-instant — updateUserById fallaba silenciosamente
Cuando un usuario existente no confirmado intentaba registrarse, la función llamaba a updateUserById() para confirmarlo. Si esa llamada fallaba, se ignoraba el error y se devolvía { exists: true } igualmente. El signInWithPassword posterior entonces fallaba sin razón aparente.

Fix: El error de updateUserById ahora se propaga al cliente como { error: message } con HTTP 400.

Eventos duplicados / inconsistentes corregidos

Doble conteo · Corregido
register_success (LoginModal) + signup_complete (useAuth) → mismo registro
Para registros por email se disparaban dos eventos distintos para el mismo acto. Además, para registros por Google OAuth no se disparaba ningún evento que el embudo SQL contabilizase, por lo que los registros por Google eran invisibles.

Fix: useAuth es la única fuente de verdad. Emite register_success (el evento del embudo) tanto para email como Google. LoginModal ya no duplica. Ventana de detección de usuario nuevo ampliada de 5 s a 10 s.

Nuevo evento de intento de registro

Nuevo
register_attempt { method: 'email' }
Se dispara antes de llamar al edge function. Permite calcular la tasa de conversión real entre intentos y registros completados, y detectar si hay un problema sistémico (muchos intentos sin éxito).
3
Tracking completo del onboarding

El flujo de onboarding de 5 pasos no disparaba ningún evento. Era imposible saber cuántos usuarios completaban cada paso, cuándo abandonaban o si aceptaban la misión propuesta. Se ha implementado cobertura total.

Eventos añadidos

onboarding_started
onboarding_step_viewed { step: 1–5 }
onboarding_mission_accepted
onboarding_mission_skipped
onboarding_completed
onboarding_skipped { from_step }

El botón "Omitir" dentro del paso 4 y el botón "Empezar misión" ahora llaman a callbacks diferenciados (onAccept / onSkip), lo que permite medir la tasa de aceptación de la primera misión propuesta por la IA.

El skip global del onboarding también incluye from_step para saber en qué punto exacto abandona cada usuario.
